# 브레인스토밍 세션

체계적으로 아이디어를 발산하고 최적의 솔루션을 선택하는 워크플로우입니다.

## 4단계 프로세스

### 1️⃣ 문제 정의 (Problem Statement)

**목표**: 해결하려는 문제를 명확히 정의

**질문**:
- 정확히 무엇이 문제인가?
- 왜 이 문제를 해결해야 하는가?
- 현재 상황은 어떠한가?
- 이상적인 결과는 무엇인가?

**프레임워크: 5W1H**
```markdown
## 문제 정의

### What (무엇)
- 문제: [구체적 문제 설명]

### Why (왜)
- 중요성: [비즈니스/기술적 이유]
- 영향: [해결 시 기대 효과]

### Who (누구)
- 이해관계자: [사용자, 팀, 고객 등]

### When (언제)
- 타임라인: [언제까지 해결해야 하는가]

### Where (어디)
- 범위: [어떤 시스템/모듈에서 발생하는가]

### How (어떻게)
- 현재 접근: [현재 어떻게 해결하려고 하는가]
```

**제약사항 식별**:
- **기술적**: 호환성, 성능, 보안
- **비즈니스**: 예산, 시간, 리소스
- **조직적**: 팀 스킬, 운영 방식

**성공 기준 정의**:
```markdown
## 성공 기준

1. **필수 (Must Have)**:
   - [ ] ...
   - [ ] ...

2. **중요 (Should Have)**:
   - [ ] ...

3. **선택 (Nice to Have)**:
   - [ ] ...
```

### 2️⃣ 아이디어 생성 (Ideation)

**원칙**: 비판 금지, 양적 발산

**브레인스토밍 규칙**:
1. 🚫 **비판 금지**: "안 될 것 같아" 금지
2. 🎯 **자유로운 발상**: 엉뚱한 아이디어도 환영
3. 📈 **양 중시**: 최소 5-10개 아이디어
4. 🔄 **결합 및 개선**: 기존 아이디어 확장

**발상 기법**:

#### A. 역발상 (Reverse Thinking)
```
문제: "사용자 이탈을 줄이려면?"
역발상: "사용자 이탈을 늘리려면?"
→ 복잡한 UI, 느린 로딩, 많은 광고
→ 해결책: UI 단순화, 성능 최적화, 광고 최소화
```

#### B. SCAMPER 기법
- **S**ubstitute (대체): 다른 것으로 바꾼다면?
- **C**ombine (결합): 결합할 수 있다면?
- **A**dapt (적용): 다른 곳에서 적용한다면?
- **M**odify (수정): 변형할 수 있다면?
- **P**ut to other uses (전용): 다른 용도로?
- **E**liminate (제거): 없앨 수 있다면?
- **R**everse (역전): 순서를 바꾼다면?

#### C. 6 Thinking Hats
- 🔴 **감정**: 직관적으로 어떻게 느껴지는가?
- ⚪ **사실**: 객관적 데이터는 무엇인가?
- 🟡 **긍정**: 장점과 기회는?
- ⚫ **부정**: 리스크와 문제점은?
- 🟢 **창의**: 새로운 아이디어는?
- 🔵 **관리**: 전체 프로세스 조율

**아이디어 템플릿**:
```markdown
## 아이디어 목록

### 아이디어 1: [제목]
- **설명**: 간단한 설명
- **장점**:
- **단점**:
- **필요 리소스**:

### 아이디어 2: [제목]
...
```

### 3️⃣ 아이디어 평가 (Evaluation)

**목표**: 각 아이디어의 실현 가능성과 효과 평가

**평가 기준**:

#### 실현 가능성 (Feasibility)
- **기술적**: 현재 기술로 구현 가능한가?
- **리소스**: 팀이 수행할 역량이 있는가?
- **시간**: 타임라인 내 완료 가능한가?
- **예산**: 비용이 적절한가?

#### 예상 효과 (Impact)
- **사용자 가치**: 사용자에게 얼마나 도움이 되는가?
- **비즈니스 가치**: 매출/성장에 기여하는가?
- **기술 부채**: 장기적으로 유지보수 가능한가?

**평가 매트릭스**:

```markdown
| 아이디어 | 실현성 | 효과 | 리스크 | 비용 | 총점 |
|---------|--------|------|--------|------|------|
| 아이디어 1 | 8 | 9 | 3 | 5 | 19 |
| 아이디어 2 | 6 | 7 | 5 | 8 | 16 |
| 아이디어 3 | 9 | 6 | 2 | 4 | 18 |

(척도: 1-10, 리스크는 낮을수록 좋음)
```

**SWOT 분석** (주요 아이디어에 적용):
```markdown
### 아이디어: [제목]

#### Strengths (강점)
-

#### Weaknesses (약점)
-

#### Opportunities (기회)
-

#### Threats (위협)
-
```

**리스크 평가**:
```markdown
## 리스크 분석

### 고위험 (즉시 대응 필요)
- [ ] 리스크 1: [설명] → 완화 방안: [...]

### 중위험 (모니터링 필요)
- [ ] 리스크 2: [설명] → 완화 방안: [...]

### 저위험 (수용 가능)
- [ ] 리스크 3: [설명]
```

### 4️⃣ 우선순위 결정 (Prioritization)

**목표**: 최선의 아이디어 선택 및 실행 순서 결정

**우선순위 프레임워크**:

#### A. Impact vs Effort Matrix

```
    높은 효과
        ↑
Quick  │  Major
Wins   │  Projects
───────┼───────── →
Fill   │  Money
Ins    │  Pits     높은 노력
```

- **Quick Wins** (빠른 승리): 우선 실행
- **Major Projects** (주요 프로젝트): 계획 후 실행
- **Fill Ins** (틈새 작업): 여유 시 실행
- **Money Pits** (돈 구멍): 지양

#### B. MoSCoW 우선순위

```markdown
## 우선순위

### Must Have (필수)
1. [아이디어 A]: 없으면 실패
2. [아이디어 B]: 핵심 기능

### Should Have (중요)
3. [아이디어 C]: 중요하지만 우회 가능
4. [아이디어 D]: 1차 출시 후 추가

### Could Have (선택)
5. [아이디어 E]: 좋지만 필수 아님

### Won't Have (제외)
- [아이디어 F]: 범위 밖
- [아이디어 G]: 리소스 부족
```

#### C. RICE Score

```
RICE = (Reach × Impact × Confidence) / Effort

- Reach: 영향받는 사용자 수
- Impact: 개인당 효과 (0.25, 0.5, 1, 2, 3)
- Confidence: 확신도 (0-100%)
- Effort: 필요 작업량 (인월)
```

**예시**:
```markdown
| 아이디어 | Reach | Impact | Confidence | Effort | RICE |
|---------|-------|--------|------------|--------|------|
| 소셜 로그인 | 5000 | 2 | 80% | 2 | 4000 |
| 푸시 알림 | 3000 | 1 | 90% | 1 | 2700 |
```

**최종 결정**:
```markdown
## 실행 계획

### Phase 1 (즉시 시작)
- [ ] 아이디어 A: [2주, 담당: 팀1]
- [ ] 아이디어 B: [1주, 담당: 팀2]

### Phase 2 (이후 진행)
- [ ] 아이디어 C: [1개월]

### Backlog (추후 고려)
- 아이디어 D
- 아이디어 E
```

## 베스트 프랙티스

### ✅ 좋은 브레인스토밍

1. **시간 제한**: 각 단계별 타임박스 (문제 정의 15분, 아이디어 30분 등)
2. **다양한 관점**: 여러 역할의 사람 참여 (개발자, 디자이너, PM 등)
3. **시각화**: 화이트보드, 포스트잇, 다이어그램 활용
4. **기록**: 모든 아이디어 문서화
5. **후속 조치**: 결정 사항을 실행 계획으로 전환

### ❌ 피해야 할 실수

1. **조기 비판**: 아이디어 생성 단계에서 비판하면 창의성 저하
2. **권위에 의존**: "사장님이 그렇게 하래요"
3. **애착**: 자기 아이디어에 집착하지 말기
4. **결정 회피**: 우선순위 없이 모든 것 시도
5. **실행 부재**: 브레인스토밍만 하고 실행 안 함

## 요약

좋은 브레인스토밍은:
- ✅ 명확한 문제 정의로 시작
- ✅ 비판 없는 자유로운 발산
- ✅ 객관적 기준으로 평가
- ✅ 실행 가능한 우선순위로 마무리

**핵심**: 발산 → 수렴 → 결정 → 실행
